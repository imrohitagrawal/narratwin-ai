# Portability Strategy

## Version

- Version: 1.0
- Stage: Stage 2 architecture, security, AI safety
- Canonical issue: `#2`
- Last updated: 2026-07-02

## Goal

NarraTwin AI must remain portable across local development, free engineering mode,
and future premium showcase mode. Users and maintainers must be able to replace
providers, export project data, regenerate derived artifacts, and run tests without
paid provider keys.

## Portability Principles

- Source documents and metadata are canonical.
- Embeddings, vector indexes, generated media, and caches are derived.
- Provider SDKs stay behind adapters.
- Provider choice is configuration-driven and recorded per run.
- Export formats use stable JSON/Markdown artifacts where possible.
- Local/dev/test does not require external services.
- Premium provider usage is optional, explicit, and auditable.

## Provider Portability

Provider interfaces:

- `LLMProvider`
- `EmbeddingProvider`
- `TranslationProvider`
- `TTSProvider`
- `STTProvider`
- `AvatarProvider`
- `SubtitleProvider`
- `VideoRenderer`
- `StorageProvider`
- `VectorStoreProvider`
- `EvaluationProvider`
- `ObservabilityProvider`

### Provider Adapter Contract

Adapter requirements:

- no provider SDK imports in domain logic
- common request/response schemas
- structured provider error codes
- contract tests for each real adapter
- mock adapter for local/dev/test
- opt-in real provider tests
- explicit provider capability metadata
- timeout and retry policy
- redaction policy for prompts, responses, traces, and errors
- provider data-retention metadata
- deterministic local fixture for each capability

Provider migration requirements:

- generated runs record provider and model metadata
- embeddings can be regenerated from canonical chunks
- evaluation results record evaluator type and version
- provider-specific artifact IDs are not exposed as canonical IDs

## Data Portability

Export bundle shape:

```text
project-export/
  manifest.json
  projects.json
  documents/
  chunks.jsonl
  walkthrough-runs.jsonl
  evaluation-results.jsonl
  context-refs.jsonl
  claim-supports.jsonl
  evidence-snapshots.jsonl
  tombstones.jsonl
  audit-events.jsonl
  artifacts/
```

Export requirements:

- include schema version
- include project ID, `tenant_id`, and synthetic local tenant ID for Stage 4
- include source document checksums
- include context references for generated outputs
- include provider metadata and estimated cost where available
- exclude provider secrets
- exclude local absolute paths
- make vector indexes optional because they can be regenerated
- include tombstones for deleted documents, chunks, runs, evaluations, and artifacts
- include synthetic local tenant and owner IDs for Stage 4 exports

### Export Tombstones

Tombstones preserve auditability after local deletion:

- `resource_type`
- `resource_id`
- `tenant_id`
- `project_id`
- `deleted_at`
- `deleted_by`
- `reason_code`
- `checksum` when available

Imports must preserve tombstones and must not resurrect deleted source content unless
the import explicitly provides a new approved document version.

Import requirements:

- validate schema version
- validate checksums where source files are present
- regenerate embeddings when vector index is missing or provider changes
- preserve run/evaluation provenance
- reject imports that would cross project or current `tenant_id` boundaries
- map `tenant_local`, `user_local`, and any validated local simulation principals
  only when importing into a trusted local workspace

## Storage Portability

Default:

- local filesystem for artifacts
- local metadata store selected in Stage 3/4
- optional JSON file snapshots enabled with `NARRATWIN_STATE_DIR` for local
  restart recovery of Stage 4 project/document/run/RAG state, Stage 6
  multilingual idempotency replay state, and Stage 7 avatar render/idempotency
  artifact metadata
- ChromaDB or equivalent local vector store behind adapter after review

Future:

- cloud object storage behind `StorageProvider`
- managed vector database behind `VectorStoreProvider`
- managed observability sink behind `ObservabilityProvider`

Rules:

- canonical IDs are generated by NarraTwin AI, not providers
- artifact paths are logical references, not absolute filesystem paths
- local JSON snapshots are a single-node durability bridge and are not an ACID,
  multi-writer, or production database substitute
- deletion and retention policies must work across storage providers
- export/import must not require the original vector store

## Runtime Portability

Local-first mode:

- no real paid provider keys required
- mock providers available
- local storage available
- deterministic fixtures for tests
- synthetic local tenant and default user are always present as `tenant_local`
  and `user_local`
- trusted local/dev/test principal simulation may use `X-Local-User-Id`; the
  header is not portable production authentication and must not be accepted as
  production identity

Future deployment mode:

- explicit environment-backed provider selection
- least-privilege secrets
- provider egress review
- dependency/security scan path
- OWASP baseline scan once web surface exists

## AI Output Portability

Generated scripts must be useful outside the original provider:

- store script text
- store context references
- store request parameters
- store evaluation result
- store provider and model metadata
- store disclosure metadata for future media

Future subtitle, audio, and video outputs must reference the grounded script and
source run rather than becoming disconnected artifacts.

## Anti-lock-in Rules

- do not store only provider-native response blobs
- do not expose provider-specific IDs as public canonical IDs
- do not require premium provider keys for tests
- do not let uploaded content select providers
- do not let provider capabilities redefine the core product contract
- do not enable research-only or non-commercial media tools by default

## Migration Scenarios

### Replace LLM Provider

Steps:

- add adapter behind `LLMProvider`
- add contract tests
- update third-party notices
- keep prompt/output schema unchanged
- record provider metadata per run

### Replace Vector Store

Steps:

- export canonical documents and chunks
- regenerate embeddings if needed
- rebuild vector index
- run retrieval parity tests
- preserve context reference identifiers or provide migration map

### Move From Local To Cloud Storage

Steps:

- add `StorageProvider` adapter
- migrate artifacts by logical ID
- preserve checksums
- update retention/deletion documentation
- add cloud-specific threat model review

### Add Avatar Provider

Steps:

- add adapter after Stage 7 approval
- verify license and provider terms
- require mock provider tests
- require consent and disclosure metadata
- update third-party notices

## Portability Validation

Before Stage 4 merge:

- project data can be represented without provider-specific SDK objects
- embeddings are treated as derived artifacts
- generated output stores context references and provider mode
- tests pass with mock/local providers

Before future release:

- project export/import is tested
- storage provider replacement path is documented
- vector-store rebuild path is tested
- provider adapter contract tests pass
- third-party notices are complete
